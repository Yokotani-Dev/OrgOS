## プリフライトチェック

### 必須チェック

- [ ] 全テストが通っている（Unit + Integration + E2E）
- [ ] ビルドが成功している
- [ ] セキュリティスキャンに問題なし
- [ ] レビュー完了（org-reviewer + org-security-reviewer）
- [ ] ステージング環境で動作確認済み
- [ ] データベースマイグレーションが検証済み
- [ ] 環境変数が本番環境に設定済み
- [ ] ロールバック手順が準備済み

### 追加チェック（該当する場合）

- [ ] API 互換性確認（Breaking changes なし）
- [ ] 外部サービス連携の確認
- [ ] キャッシュ無効化の手順確認
- [ ] DNS / CDN 設定の確認
- [ ] SSL 証明書の有効期限確認

## デプロイ手順

### Step 1: 事前準備

1. メンテナンスウィンドウの通知（該当する場合）
2. データベースバックアップの取得
3. 現在のバージョンを記録（ロールバック先）

### Step 2: デプロイ実行

[デプロイコマンドまたは手順]

### Step 3: 検証

#### 即時確認（デプロイ後 5 分以内）

- [ ] アプリケーションが応答する（ヘルスチェック）
- [ ] ログにエラーが出ていない
- [ ] 主要 API エンドポイントが正常応答

#### 短期確認（デプロイ後 30 分以内）

- [ ] 主要ユーザーフローが動作する
- [ ] パフォーマンスメトリクスが正常範囲
- [ ] エラーレートが増加していない

#### 中期確認（デプロイ後 24 時間以内）

- [ ] ユーザーからの問題報告がない
- [ ] メトリクスが安定している
- [ ] 非同期ジョブが正常実行されている

## ロールバック手順

### トリガー条件

以下のいずれかに該当した場合、即座にロールバック:

| 条件 | 判断 |
|------|------|
| アプリケーションが応答しない | 即時ロールバック |
| エラーレートが 5% を超える | 即時ロールバック |
| 主要フローが動作しない | 即時ロールバック |
| パフォーマンスが 50% 以上劣化 | 10 分以内に判断 |
| 軽微な問題が複数報告 | 30 分以内に判断 |

### ロールバック手順

1. 前のバージョンにデプロイ
2. データベースマイグレーションのロールバック（該当する場合）
3. キャッシュのクリア
4. 動作確認
5. DECISIONS.md にロールバック理由を記録
6. 修正タスクを TASKS.yaml に追加

## デプロイタイミング

### 推奨

| タイミング | 推奨度 | 理由 |
|-----------|--------|------|
| 火〜木 午前 | 推奨 | 問題発生時に対応可能 |
| 月曜 午前 | 可 | 週の初めに問題を検出 |
| 金曜 午後 | 非推奨 | 週末に問題が発覚するリスク |
| 深夜・休日 | 禁止 | 対応者が不在 |

### 例外

- **ホットフィックス**: セキュリティ脆弱性や重大なバグは曜日・時間を問わず即座にデプロイ
- **メンテナンスウィンドウ**: 事前に通知した上で、ユーザー影響が最小の時間帯に実施

## 段階的ロールアウト

### Canary デプロイ

```
Step 1: トラフィックの 5% に新バージョンをデプロイ
  → 15 分間メトリクスを監視
  → エラーレート増加なし → Step 2 へ
  → エラーレート増加 → ロールバック

Step 2: トラフィックの 25% に拡大
  → 30 分間メトリクスを監視
  → 問題なし → Step 3 へ

Step 3: トラフィックの 100% に拡大
  → 24 時間監視
```

### Feature Flag

```typescript
// 新機能を Feature Flag で制御
if (featureFlags.isEnabled('new-checkout', { userId })) {
  return <NewCheckout />;
}
return <OldCheckout />;
```

Feature Flag の利点:
- デプロイとリリースを分離
- 問題発生時に Flag を OFF にするだけ（ロールバック不要）
- A/B テストに活用可能

## OrgOS RELEASE ステージとの統合

### /org-tick での自動チェック

RELEASE ステージの /org-tick で以下を自動実行:

```
1. プリフライトチェックの確認
   → 全チェックが通っていなければ blocked

2. ロールアウト計画の存在確認
   → .ai/DESIGN/ROLLOUT_PLAN.md がなければ作成を提案

3. Owner 承認の確認
   → CONTROL.yaml の allow_deploy が true か確認
   → false なら Owner に承認を依頼
```

### CONTROL.yaml との連携

```yaml
# CONTROL.yaml
allow_deploy: false  # Owner が true に変更するまでデプロイ不可

gates:
  release_approved: false  # リリースゲート
```
