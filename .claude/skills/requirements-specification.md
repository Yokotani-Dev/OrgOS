# 要件仕様スキル

> BRIEF.md から構造化された仕様書（SPEC）を作成するスキル

> **参照元**: GitHub awesome-copilot create-specification / prd skills

---

## Iron Law

> 要件定義の鉄則。例外なし。

1. **acceptance に「TBD」「後で決める」を書かない** - 検証できない条件は条件ではない
2. **全ての P0/P1 要件に REQ-ID を付与する** - ID のない要件は追跡できない

---

## このスキルの位置づけ

OrgOS ワークフローにおける仕様書の位置:

```
BRIEF.md（やりたいこと）
  ↓ /org-brief で収集
PROJECT.md（要件整理）
  ↓ このスキルで構造化
.ai/DESIGN/SPEC.md（仕様書）  ← 新設
  ↓ タスク分解
TASKS.yaml（実行計画）
```

**なぜ仕様書が必要か**:
- 要件に ID が付く → テストと紐付け可能（REQ-001 → TEST-001）
- 受入基準が明確 → 「完了」の定義が曖昧にならない
- スコープクリープの早期検出 → 仕様にない機能追加を防ぐ

## 要件 ID のルール

### ID 体系

| プレフィックス | 意味 | 例 |
|-------------|------|-----|
| REQ | 機能要件 | REQ-001: ユーザーがログインできる |
| NFR | 非機能要件 | NFR-001: レスポンスタイム < 3秒 |
| CON | 制約 | CON-001: PostgreSQL を使用 |
| SEC | セキュリティ要件 | SEC-001: パスワードはハッシュ化 |

### TASKS.yaml との紐付け

```yaml
# TASKS.yaml
- id: T-003
  title: "ユーザーログイン機能の実装"
  deps: ["T-002"]
  spec_refs: ["REQ-001", "REQ-002", "SEC-001"]  # 仕様書への参照
  acceptance:
    - "REQ-001: Given 未認証ユーザー, When 正しい認証情報を送信, Then ログイン成功"
    - "REQ-002: Given ログイン済みユーザー, When トークン期限切れ, Then リフレッシュトークンで更新"
```

## 受入基準の書き方

### Given-When-Then パターン

```markdown
// ❌ 悪い例: 曖昧な受入基準
- ユーザーがログインできること
- エラー時に適切なメッセージを表示

// ✅ 良い例: 具体的な Given-When-Then
REQ-001: ログイン機能
- Given: 未認証のユーザーがログインページを開いている
- When: 正しいメールアドレスとパスワードを入力し「ログイン」をクリック
- Then: ダッシュボードにリダイレクトされ、ユーザー名が表示される

REQ-001-ERR: ログインエラー
- Given: 未認証のユーザーがログインページを開いている
- When: 存在しないメールアドレスを入力し「ログイン」をクリック
- Then: 「メールアドレスまたはパスワードが正しくありません」とインライン表示される
- And: パスワードフィールドがクリアされる
```

### 測定可能な基準

```markdown
// ❌ 悪い例: 測定不可能
- ページが速く表示される
- ユーザーにとって使いやすい

// ✅ 良い例: 測定可能
- NFR-001: ページ初期表示が 3 秒以内（LCP < 3s）
- NFR-002: Lighthouse アクセシビリティスコア 90 点以上
- NFR-003: 同時接続 100 ユーザーでレスポンスタイム p95 < 500ms
```

## 優先度の定義

| 優先度 | 定義 | 例 |
|--------|------|-----|
| **P0** | 必須（これがないとリリースできない） | ログイン機能、決済機能 |
| **P1** | 重要（リリース時にあるべき） | パスワードリセット、メール通知 |
| **P2** | あると良い（後回し可能） | ダークモード、エクスポート機能 |
| **P3** | 将来検討 | モバイルアプリ対応、AI 機能 |

## OrgOS ステージとの統合

### REQUIREMENTS ステージで実行

```
1. BRIEF.md + PROJECT.md を読む
2. 要件を REQ-XXX 形式で構造化
3. 受入基準を Given-When-Then で記述
4. 非機能要件を NFR-XXX で定義
5. 制約を CON-XXX で定義
6. .ai/DESIGN/SPEC.md に保存
7. DECISIONS.md に仕様決定を記録
```

### DESIGN ステージへの引き継ぎ

仕様書の REQ-XXX が DESIGN ステージの入力になる:
- API 設計（REQ → エンドポイント定義）
- データモデル設計（REQ → スキーマ定義）
- テスト計画（REQ → テストケース）

### TASKS.yaml への分解

各 REQ を 1 つ以上のタスクに分解:
- REQ-001 → T-003（実装）+ T-004（テスト）
- タスクの `acceptance` に Given-When-Then を転記
- `spec_refs` で仕様書への逆参照を維持

## チェックリスト

仕様書完成時に確認:

- [ ] すべての要件に ID が付いている（REQ-XXX）
- [ ] すべての要件に優先度が設定されている
- [ ] すべての P0/P1 要件に Given-When-Then の受入基準がある
- [ ] 非機能要件が測定可能な値で定義されている
- [ ] スコープの「含まない」が明記されている
- [ ] 用語定義が記載されている
- [ ] TASKS.yaml のタスクが spec_refs で仕様書を参照している
