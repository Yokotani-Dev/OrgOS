# 合理化防止ルール

> AI エージェントがルールを「知っているのに従わない」問題を防ぐ仕組み

> **参照元**: obra/superpowers rationalization prevention system

---

## 基本原則

**ルールの文面に違反することは、ルールの精神に違反することである。**

「この場合は例外」「今回だけ」「ほぼ同じだから」は合理化（言い訳）である。
ルールを破る正当な理由がある場合は、Owner に確認してから破る。

---

## Iron Law（鉄則）

各スキル・ルールには破ってはいけない鉄則がある。鉄則に「例外」はない。

### OrgOS の鉄則一覧

| 領域 | Iron Law | 違反例 |
|------|----------|--------|
| **タスク管理** | 全作業を TASKS.yaml に登録してから実行 | 「小さいからそのまま直す」 |
| **TDD** | テストなしの本番コードは禁止 | 「後でテスト書く」 |
| **レビュー** | Implementer の自己報告を信用しない | 「完了と言っているからOK」 |
| **セキュリティ** | シークレットをコードにハードコードしない | 「テスト用だから」 |
| **デプロイ** | Owner 承認なしに本番デプロイしない | 「緊急だから」 |
| **スコープ** | スコープ外の依頼はOwner確認なしに実行しない | 「関連しているから」 |
| **ファイル保護** | 保護対象ファイルを無断で変更しない | 「小さな修正だから」 |

---

## 言い訳テーブル

AI が使いがちな言い訳と、正しい対応:

| 言い訳 | なぜダメか | 正しい対応 |
|--------|-----------|-----------|
| 「この場合は例外」 | 例外は Owner が判断する | Owner に確認する |
| 「時間がないから省略」 | 時間は AI が管理する量ではない | ルール通りに実行する |
| 「ほぼ同じだから」 | 「ほぼ」は「違う」 | 正確にルールに従う |
| 「後でやる」 | 後は来ない | 今やる |
| 「小さい変更だから」 | 規模に関係なくルールは適用 | TASKS.yaml に登録する |
| 「テスト環境だから」 | テストコードにもルールは適用 | 本番と同じ基準で書く |
| 「既存コードがこうだから」 | 既存の問題は新たな問題の正当化にならない | 既存の問題は別タスクで修正 |
| 「完了と報告があった」 | 報告を信用するな | 自分で検証する |
| 「Owner が急いでいる」 | 品質を犠牲にする許可ではない | 品質を保ったまま最速で進む |
| 「一時的な対応」 | 一時的なものは永続化する | 恒久対応を最初から行う |

## Red Flags（今すぐ止まれ）

以下の状況を検出したら、即座に作業を停止して振り返る:

### 作業フロー

- [ ] TASKS.yaml に登録せずに作業を始めようとしている
- [ ] テストを書かずに実装コードを書こうとしている
- [ ] レビューをスキップしようとしている
- [ ] 「ちょっとした修正」を繰り返している

### コード品質

- [ ] エラーハンドリングを省略しようとしている
- [ ] 型定義を any にしようとしている
- [ ] console.log を本番コードに残そうとしている
- [ ] コメントアウトでコードを残そうとしている

### セキュリティ

- [ ] 環境変数の代わりにハードコードしようとしている
- [ ] 入力バリデーションを省略しようとしている
- [ ] HTTPS 以外で通信しようとしている

### プロセス

- [ ] Owner の承認なしにスコープを拡大しようとしている
- [ ] DECISIONS.md に記録せずに設計判断をしようとしている
- [ ] エージェントの自己報告を検証せずに信用しようとしている

## 適用方法

### Manager の自己チェック

各 Tick の開始時に以下を確認:

```
1. 前の Tick で Iron Law に違反していないか？
2. 言い訳テーブルの言い訳を使っていないか？
3. Red Flags に該当する状況はないか？
```

### エージェント起動時

サブエージェントに指示を出す際、以下を含める:

```
Iron Law: [該当する鉄則]
報告形式: 完了の自己報告だけでなく、検証可能な証拠を含めること
```

### レビュー時

org-reviewer は以下を追加チェック:

```
- 実装がタスクの spec_refs を全てカバーしているか（自己報告を信用しない）
- テストが受入基準の全ケースをカバーしているか
- スコープ外の変更が含まれていないか
```

---

## スキルへの Iron Law 追加ガイド

各スキルの冒頭に Iron Law セクションを追加する:

```markdown
## Iron Law

> このスキルの鉄則。例外なし。

1. [鉄則1]
2. [鉄則2]
```

Iron Law の条件:
- 違反すると取り返しのつかない問題が起きる
- 「この場合は例外」が成立しない
- テストで違反を検出可能
