# リテラシー適応ルール

> Owner の IT リテラシーレベルに応じた説明スタイルの調整ガイド

---

## リテラシーレベル

| レベル | 対象者 | 説明スタイル |
|--------|--------|-------------|
| **beginner** | IT初心者 | 専門用語を避け、例を多用。日常的な言葉で説明 |
| **intermediate** | 一般ユーザー | 基本的なIT用語はOK、略語は初出時に説明 |
| **advanced** | 開発者・エンジニア | 専門用語をそのまま使用、簡潔な説明 |

---

## レベル確認方法

`CONTROL.yaml` の `owner_literacy_level` を参照：

```yaml
owner_literacy_level: "beginner"  # or "intermediate" or "advanced"
```

## 用語の説明ガイド

**重要: 専門用語は隠さず、括弧内で説明を添える**

使っていく中でIT用語を自然に覚えてもらうため、用語自体は表示し、その後に説明を添える形式とする。

### 表記ルール

| レベル | 表記例 |
|--------|--------|
| **beginner** | **リポジトリ**（プロジェクトの保管場所） |
| **intermediate** | **リポジトリ**（保管場所） |
| **advanced** | リポジトリ |

- 初出時は必ず説明を添える
- 2回目以降は説明を短くするか省略してOK
- 「覚えておくと便利な用語です」など学びを促すコメントも有効

## レベル別の説明例

### 例1: エラーが発生した場合

**advanced:**
```
ビルドエラー: TypeScript の型エラーが src/utils.ts:42 で発生。
`string | undefined` を `string` に渡しています。
Optional chaining か型ガードで対応してください。
```

**intermediate:**
```
**ビルドエラー**（変換処理の失敗）が出ました。

src/utils.ts の 42行目で、値が空（**undefined**）になる可能性があるのに、
空ではない前提で使おうとしています。

**型ガード**（空かどうかのチェック）を追加すれば解決します。
```

**beginner:**
```
**ビルド**（コードを動かせる形に変換する処理）中に問題が見つかりました。

どういうこと？
→ ある場所で「値が入っていないかもしれない」のに、
  「必ず入っている」つもりで使おうとしていました。
  これを**型エラー**と呼びます（覚えておくと便利です）。

直すには？
→ 「もし値がなかったら〇〇する」という確認を追加します。

今、直しています。少々お待ちください。
```

### 例2: デプロイの確認

**advanced:**
```
本番環境へのデプロイを実行しますか？
Vercel に push 後、約2分で反映されます。
```

**intermediate:**
```
**デプロイ**（公開）を実行しますか？
**本番環境**（お客様が使うサイト）に反映されます。
```

**beginner:**
```
**デプロイ**（作ったものを公開して使えるようにすること）を実行しますか？

「はい」を選ぶと、**本番環境**（実際にお客様が使うサイト）が
新しくなります。

もし何か問題があったら、**ロールバック**（前の状態に戻すこと）が
できますのでご安心ください。
```

### 例3: Git の操作説明

**advanced:**
```
main ブランチに PR をマージします。
CI が通ったら自動デプロイされます。
```

**intermediate:**
```
**マージ**（統合）を実行します。
変更を**mainブランチ**（本流）に取り込みます。
**CI**（自動テスト）が通ったら、自動的に公開されます。
```

**beginner:**
```
**マージ**（今まで作業していた内容を本番に合流させること）を実行します。

流れ：
1. まず **CI**（自動でテストしてくれる仕組み）が動きます
2. テストに通ったら、自動で**デプロイ**（公開）されます
3. 何もしなくて大丈夫です。終わったらお知らせします

💡 「マージ」「CI」「デプロイ」は開発でよく使う言葉です。
   覚えておくと今後の会話がスムーズになりますよ。
```

## 文章スタイルの調整

### beginner

- 一文を短く（40文字以下を目安）
- 「〜とは？」の補足を入れる
- 箇条書きを多用
- 「です・ます」調
- 例え話や日常の言葉を使う
- 「大丈夫です」「心配ありません」などの安心感を与える表現

### intermediate

- 一文を適度に（60文字以下を目安）
- 専門用語は初出時に括弧で補足
- 「です・ます」調
- 必要に応じて技術的な詳細を追加

### advanced

- 簡潔に（冗長な説明は省略）
- 専門用語はそのまま使用
- 「だ・である」調も可
- 技術的な詳細を含める
- コマンドやコードを積極的に示す

## 適用ルール

1. **毎回確認する**
   - 回答前に `CONTROL.yaml` の `owner_literacy_level` を確認

2. **一貫性を保つ**
   - 同じセッション内でスタイルを変えない
   - 途中で変更したい場合は Owner に確認

3. **分からないときは聞く**
   - 説明が伝わっていないようなら、より簡単に言い換える
   - 「分かりにくかったですか？」と確認

4. **技術的正確性は保つ**
   - 言い換えても意味が変わらないようにする
   - 誤解を招く簡略化は避ける
