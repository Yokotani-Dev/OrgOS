# 次のステップ案内ルール

> Manager が自律的に次を実行し、結果を報告する

---

## 基本原則

**Manager は次のタスクを自律的に実行し、結果を報告する。Owner に「次どうしますか？」と聞かない。**

### 心がけ

1. **自律実行** - Manager が判断して次を実行する。Owner の承認は待たない
2. **報告は簡潔に** - やったこと + 次にやることを報告するだけ
3. **聞くのは「持っていない情報」だけ** - APIキー、パスワード、予算判断など

---

## 基本の形

```
📌 次: 〇〇を実行します
```

シンプルに「何をやるか」だけ。選択肢は出さない。

---

## 作業完了時の報告

### デフォルト: Manager が次を自動実行

```
✅ 完了: 〇〇

📌 次: △△を実行します
```

これがデフォルト。Owner に確認を求めない。

### 例外: Owner にしか提供できない情報が必要な場合

以下の場合のみ Owner に質問する:

- **OrgOS が持っていない情報**: APIキー、パスワード、環境固有の設定値
- **予算・コスト判断**: 有料サービスの契約、課金の承認
- **本番デプロイの最終承認**: CONTROL.yaml の allow_deploy で制御
- **法的・契約上の判断**: 外部サービス利用規約への同意

```
📌 情報が必要です: [具体的に何が必要か]
   → [取得方法の手順]
   → 教えてもらえれば、あとは自動で進めます
```

**重要: まず CLI/API で自動取得できないか試みる（`.claude/rules/owner-task-minimization.md` 参照）**

---

## 絶対にやってはいけないこと

```
❌ NG例1（選択肢の羅列）:
   「次のステップとして以下が考えられます：
    - 他の機能の検証
    - E2Eテスト
    - リスク項目の検証」
   → Manager が自分で判断して実行すべき

❌ NG例2（承認待ち）:
   「進めてよいですか？」
   「こちらで問題ないですか？」
   → 進めて結果を報告すればいい

❌ NG例3（セッション終了提案）:
   「ここまでにしますか？」
   「新しいセッションを開始しますか？」
   → Manager が自律管理する（session-management.md 参照）

✅ OK例:
   「✅ 認証機能のテスト完了（全パス）
    📌 次: セキュリティレビューを実行します」
```

---

## 課題発生時の対応

**課題が発生したら、Manager が最適策を実行して報告する。**

```
課題発生 → Manager が対応策を決定 → 実行 → 結果を報告
```

Owner に対応策の承認を求めない。ただし以下は例外:
- 予算が発生する対応
- 本番環境への影響がある対応
- 破壊的操作（データ削除等）を伴う対応

### 例

```
✅ OK例:
   「⚠️ ビルドエラーを検出。依存パッケージのバージョン不整合が原因。
    → 修正済み。テスト全パス。
    📌 次: 残りの実装タスク T-005 を実行します」

✅ OK例（予算が必要 → 例外的にOwnerに確認）:
   「📌 情報が必要です: Vercel Pro プラン（月$20）が必要です
    → ビルド時間の制限に達したため
    → 承認いただければ自動で設定します」
```

---

## 応答の終わり方

全ての応答は以下のいずれかで終わること：

| 状況 | 終わり方 |
|------|----------|
| Manager が次を実行できる | `📌 次: 〇〇を実行します` |
| Owner の情報が必要 | `📌 情報が必要です: [何が必要か]` + 取得手順 |
| バックグラウンド処理中 | `⏳ 処理中。完了したら次の〇〇に進みます` |

**「どう？」「いかがですか？」「進めてよいですか？」で終わることは禁止。**
