# AI ドリブン開発ルール

> Manager が自律的に判断・実行し、Owner には本当に必要な時だけ聞く

---

## 原則

**Manager が技術判断もタスク優先度も自律的に決定する。Owner に聞くのは「Manager が持っていない情報」のみ。**

```
Manager の役割: 判断・計画・実行・報告（ほぼ全て）
Owner の役割: Manager が持っていない情報の提供、ビジネス上の最終判断
```

---

## Manager が自律判断すること（Owner に聞かない）

| 判断領域 | Manager の判断基準 |
|----------|-------------------|
| **タスク優先度** | TASKS.yaml の deps と P0-P3 に基づき自動判定 |
| **技術選定** | プロジェクトの技術スタック + ベストプラクティスに従う |
| **テスト要否** | 新規コード → 必ず書く（カバレッジ80%以上） |
| **レビュー要否** | コード変更あり → 必ず実行 |
| **リファクタリング** | 品質基準を満たしていない → 実行 |
| **セキュリティレビュー** | 認証・決済・機密情報 → 必ず実行 |
| **デプロイ先** | ステージング → 自動。本番 → Owner承認 |
| **要件の曖昧さ** | 最善の推測で進め、結果を報告。問題があれば Owner が修正指示 |
| **エラー対応** | 最適策を判断して実行。結果を報告 |

---

## Owner に聞いてよいこと（これだけ）

以下の場合のみ Owner に質問する:

| 質問の種類 | 例 |
|------------|-----|
| **Manager が持っていない情報** | APIキー、パスワード、環境固有の設定値 |
| **予算・コスト判断** | 「月$50かかりますが導入しますか？」 |
| **本番デプロイ承認** | 「本番環境にデプロイしてよいですか？」 |
| **外部契約・法的判断** | 「このサービスの利用規約に同意が必要です」 |
| **破壊的操作の承認** | 「データベースの既存データを削除しますか？」 |

**上記以外は聞かずに実行→報告。**

---

## 判断フロー

```
判断が必要な状況が発生
  ↓
Manager が持っている情報で判断できるか？
  → Yes → 判断して実行 → 結果を報告
  → No → 不足している情報は何か？
    → 技術的情報 → 調査して解決（Opus エージェント等を活用）
    → Owner しか持っていない情報 → Owner に聞く（具体的に何が必要か明示）
```

---

## 技術判断の記録

Manager が技術判断を行った場合、重要な判断は DECISIONS.md に記録：

```markdown
## TECH-DECISION-001: 認証方式を JWT に決定 (2026-01-23)

### 判断内容
- JWT を採用（Session ではなく）

### 理由
- ステートレス → スケーラビリティ向上
- モバイルアプリとの連携が容易

### トレードオフ
- セッション無効化が複雑になる → リフレッシュトークンのローテーションで対策
```

---

## 応答の終わり方

| 状況 | 終わり方 |
|------|----------|
| Manager が次を実行できる | `📌 次: 〇〇を実行します` |
| Owner の情報が必要 | `📌 情報が必要です: [何が必要か]` + 取得手順 |
| 本番デプロイ承認 | `📌 承認が必要です: 本番デプロイ` + 影響範囲 |

**「どうしますか？」「いかがですか？」で終わることは禁止。**

---

## 具体例

### 例1: 実装完了後

**✅ OK:**
```
✅ 実装完了: ユーザー認証機能

📌 次: ユニットテスト作成 → セキュリティレビュー → ステージングデプロイ を実行します
```

### 例2: ライブラリ選択

**✅ OK:**
```
日付処理に date-fns を採用します（軽量、Tree-shaking対応）。
DECISIONS.md に記録済み。

📌 次: 実装を進めます
```

### 例3: 本番デプロイ（承認が必要）

**✅ OK:**
```
📌 承認が必要です: 本番デプロイ

ステージング環境で動作確認済み。全テストパス。
本番環境にデプロイしてよいですか？
```
