## Context

T-OS-324 で proposal 生成が稼働。本タスクで「proposal を実際に apply する engine」を作る。

4 段階 rollout (Phase 2 SYNTHESIS §3):
- **shadow**: ファイル変更せず、ログのみ
- **canary**: 1 件のみ apply、24h 監視
- **progressive**: 段階的に apply、各段階で eval pass を確認
- **full**: 全 apply

autonomy_level マッピング (authority-layer.md):
- silent_execute → shadow → 即 full
- execute_with_report → canary 24h → 自動 full
- ask_before_execute → canary → Owner approval 待ち
- owner_only → 提案のみ、apply は Owner 手動

## Acceptance Criteria

### A1: application-record.yaml schema
JSON Schema 2020-12 準拠。各 record:
- record_id (AR-YYYY-MM-DD-NNN)
- proposal_ref (P-XXX-XXX-XXX)
- target_file
- before_checksum, after_checksum
- rollout_stage (shadow / canary / progressive / full)
- autonomy_level_at_apply
- applied_at, applied_by (always "system" for auto-apply)
- eval_results (pre + post)
- rollback_ref (null or RB-YYYY-MM-DD-NNN)
- iron_law_check (forbidden file への apply は record も作らない)

### A2: apply.sh
- `bash scripts/evolution/apply.sh <proposal_id> [--stage shadow|canary|progressive|full]`
- proposal を読み、target_file への変更を rollout_stage に応じて実行
- iron_law_check: forbidden file (`.claude/rules/rationalization-prevention.md` 等) なら reject
- record を `.ai/EVOLUTION/applied/<record_id>.yaml` に保存
- shadow stage: 実ファイル変更なし、log のみ
- canary stage: 実ファイル変更 + 監視 marker 設置 (24h 後に scheduled job が確認、本タスク範囲外)

### A3: rollback.sh
- `bash scripts/evolution/rollback.sh <record_id>`
- record を読み、before_checksum 復元
- consecutive_revert カウンタ管理 (3 連続 revert で apply.sh 全停止)
- rollback_ref を application-record に back-write

### A4: 検証
- shadow apply で実ファイル変更なし
- canary apply で record 生成
- iron_law_check で forbidden file が reject
- rollback で before 状態に戻る
- consecutive_revert=3 で apply.sh が停止 (test fixture)
- shellcheck 通過
