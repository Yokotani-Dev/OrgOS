# Work Order: T-OS-300 — OrgOS 課題棚卸し + 改善案 + 機能案 (Codex 独立分析)

## Task
- ID: T-OS-300 (Codex 側分析)
- Title: OrgOS 全体の課題と改善余地を独立視点で網羅的に洗い出す
- Role: reviewer (read-only 分析)
- Priority: P1

## Allowed Paths (write)
- `.ai/REVIEW/T-OS-300/CODEX_ANALYSIS.md` (新規作成)
- `.ai/CODEX/RESULTS/T-OS-300.md` (報告書)

すべてのその他のパスは **read-only**。コード/設定/ルールは絶対に編集しないこと。

## Context

OrgOS は Claude Code を Chief of Staff として継続稼働させる OS レイヤーで、現在 Phase 1-5 (Memory / Optionality / Intake / Handoff / Authority) の基盤実装が完了し、Manager Quality Eval も 20/20 達成済み。一方で、複雑性の増加、ルールの重複、運用負荷、評価の自動化欠落、ポータビリティ等で潜在課題があると Owner は感じている。

Owner からの依頼:
> claudeとcodexでorgOSの課題を棚卸してほしい。課題だけじゃなくて、これがより価値を発揮するような改善案、機能案も合わせて洗い出して。時間をかけていいからジックリ洗い出して欲しい

Claude 側でも独立に同等の分析が走ります。両者を Manager が統合します。Codex は **独立性が価値**なので、Claude 視点に引きずられず、コード/スクリプト/スキーマ/運用フロー観点で深掘りしてください。

## Owner からの追加観点 (重要・必ず深掘り)

> 「フォルダ構成がぐちゃぐちゃ。ガバナンスが効いていない。OrgOS しか触らないところとユーザーが触るフォルダをまず分けて、その上で、ちゃんとわかりやすいフォルダ体系にしてほしい。outputs とかが時系列で出てくるのもあまり意味ない。目的・分類ごとのフォルダ分けが理想。」

これを最重要テーマとして扱う。具体的に以下を出力:

- 現状フォルダの **責務分類**: (a) OrgOS 内部のみ / (b) Owner が触る / (c) AI 生成物 / (d) 一時/キャッシュ / (e) 不明・混在
- **混在/責務不明** ファイル/ディレクトリの実証 (path 単位)
- **時系列ベース** (例: outputs/YYYY-MM-DD) を **目的・分類ベース** に置き換える具体案
- 推奨される **3 階層以内** のディレクトリツリー (mermaid 可)
- 移行計画 (破壊的変更を最小化、ハードリンク or symlink で互換維持等)
- README / ガイド配置と命名規則

**この観点単体で 1 セクション (Folder Governance) を作る**。

## Owner からの追加観点 2 (重要・必ず深掘り)

> 「GitHub からクローンした際に `/org-start` を押してから起動するまで時間がかかるのも悩み。もっとユーザー体験を向上させたい。」

`/org-start` を含む onboarding 体験を **計測 + 分解 + 短縮** する観点で深掘り:

- `/org-start` の各ステップ所要時間の推定/計測 (.claude/commands/org-start.md と関連 hook を読み解く)
- ボトルネック特定 (大きい台帳の Read, scan.sh の I/O, hook の重複, 順次実行)
- 並列化/遅延ロード/キャッシュ余地 (例: capability scan は初回非同期、bootstrap は最小台帳のみ)
- **First Meaningful Response までの時間 (FMR)** 短縮目標値 (現状仮 X 秒 → 目標 Y 秒)
- 既存 hook (SessionStart) の改善余地
- 「インストール → 最初のタスクが進む」までの ZTI (Zero-to-Insight) 体験再設計
- 進捗インジケータ (今何を読んでいるか) の有無と追加価値

**この観点単体で 1 セクション (Onboarding & Startup UX) を作る**。



## Analysis Scope

以下 12 領域を最低限カバー (順不同):

1. **ルール層** (`.claude/rules/` 23 ファイル) — 重複/矛盾/抽象度/Iron Law の実効性
2. **エージェント層** (`.claude/agents/`) — 役割分離、責務、CODEX_WORKER_GUIDE との整合
3. **スキル層** (`.claude/skills/`) — 利用頻度、CSO 原則 (description 短文化) の遵守
4. **スクリプト層** (`scripts/`) — bootstrap/scan/eval/memory lint/authority/session 系の品質、エラー処理、idempotency
5. **台帳層** (`.ai/`) — schema 整合、参照関係、stale データ、scale 限界 (TASKS.yaml 1394 行等)
6. **評価層** (`.claude/evals/manager-quality/`) — 6 指標 × 20 ケースの妥当性、回帰検出、CI 連携余地
7. **委譲プロトコル** (`handoff-protocol.md` + schema) — 実運用での記録充足、retry policy、quarantine 動作
8. **Authority Layer** (`authority-layer.md` + 3 schemas) — runtime enforcement の実装度、approval workflow の使い勝手
9. **Memory Lifecycle** (`memory-lifecycle.md`) — secret pointer 強制、scope/transferability の実用性
10. **Owner UX** — bootstrap, OWNER_INBOX/COMMENTS, slash commands, suggest-next.sh の実体験
11. **クロスプロジェクト・ポータビリティ** — `/org-import`, `/org-publish`, GitHub への展開
12. **観測性 / デバッグ性** — エラー時の root cause analysis, telemetry, logging
13. **フォルダ構成・ガバナンス** — Owner 追加観点 1 (上記参照)。最重要テーマ
14. **Onboarding & Startup UX** — Owner 追加観点 2 (上記参照)。最重要テーマ

## Deliverables

`.ai/REVIEW/T-OS-300/CODEX_ANALYSIS.md` に以下を構造化して書く:

### 1. Executive Summary (5 行以内)
全体評価、最も致命的な 3 課題、最もインパクトある 3 改善案。

### 2. Issues (課題)
形式: `ISSUE-CDX-XXX | severity: P0/P1/P2 | category | title | evidence (file:line) | impact | proposed fix`

最低 20 件、できれば 30+ 件。仮説でなく **コード/ファイル証拠** で示す。

### 3. Improvements (既存機能の改善案)
形式: `IMP-CDX-XXX | priority: P0/P1/P2 | area | proposal | rationale | effort (S/M/L) | risk`

最低 15 件。

### 4. New Features (新機能案 — OrgOS の価値を更に引き上げる)
形式: `FEAT-CDX-XXX | priority | area | proposal | value hypothesis | dependency | effort`

最低 10 件。

### 5. Removal/Simplification Candidates (削除/簡素化候補)
冗長/未使用/過剰設計の検出。最低 5 件。

### 6. Folder Governance (最重要セクション)
6.1 現状フォルダ責務マトリクス (Owner触る / OrgOS専用 / AI生成 / 一時)
6.2 混在/責務不明の実証 (具体パス)
6.3 outputs/ の時系列 → 目的別への再設計
6.4 推奨ディレクトリツリー (3 階層以内、mermaid 可)
6.5 移行計画 (破壊度最小、互換性維持戦略)
6.6 命名規則 + README 配置

### 7. Onboarding & Startup UX (最重要セクション)
7.1 `/org-start` の現状所要時間分解 (推定可)
7.2 ボトルネック (証拠つき)
7.3 並列化 / 遅延ロード / キャッシュ提案
7.4 FMR (First Meaningful Response) 目標値設定
7.5 ZTI (Zero-to-Insight) 体験設計図
7.6 進捗可視化と中断/再開耐性

### 8. Cross-cutting Themes (横断テーマ)
領域横断の構造的問題と systemic な対処方針。3-5 テーマ。

### 9. Suggested Task IDs
T-OS-301 以降の形式で、実装タスクの粒度に分解した提案 (P0 のみ、最大 10 件)。

## Instructions

1. 必ず複数ファイルを Read してから判断する。命名や docstring から推測しない。
2. **Iron Law 違反の証拠**を優先的に探す (例: rule に書いてあるが script では未実装, schema にあるが実体ファイルに反映されていない 等)。
3. Claude 視点に寄らない**コード/スキーマ/スクリプト品質**の指摘を歓迎。
4. 機能案は「Owner の認知負荷を下げるか」「自律性を上げるか」「観測可能性を上げるか」の 3 軸で評価する。
5. 並列実行で他に Claude 分析が走っているため、レポートは **独立性** を優先。Claude が見そうな観点 (UX/読みやすさ/語彙統一) はあえて触れず、コード/構造/実行可能性に集中してよい。
