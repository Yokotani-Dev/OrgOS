## Context

### 背景
Pro 指摘: v1 の CAPABILITIES.yaml は「使える CLI 一覧」止まり。
**tool manifest** として以下が必要:
- auth_status（認証済みか）
- risk_level（実行リスク）
- supports_dry_run（dry-run 可否）
- owner_approval_required_for（承認必要操作のリスト）
- common_operations（再利用可能な操作パターン）
- input_resolution_order（必要な入力の解決順序）

### 解決する問題

- **capability_reuse_rate**: baseline 0% を > 80% に押し上げる基盤
- **MCP 互換**: 将来の MCP サーバー統合との相互運用性

## Acceptance Criteria

### A1: CAPABILITIES.yaml schema 定義

`.claude/schemas/capability.yaml`:

```yaml
capabilities:
  - id: string                    # cli_<tool> | api_<service> | mcp_<server> | internal_<name>
    kind: enum[cli, api, mcp, script, internal_skill, internal_agent]
    name: string                  # 表示名
    command: string | null        # CLI の場合のコマンド名（null if kind != cli）
    path: string | null           # フルパス（検出時に埋める）
    version: string | null
    status: enum[available, unavailable, degraded, unknown]
    auth_status: enum[verified, unverified, expired, not_required, unknown]
    verified_at: date | null
    risk_level: enum[low, medium, high, critical]
    supports_dry_run: bool
    owner_approval_required_for:
      - operation: string         # "delete", "production_deploy", "billing", etc.
        reason: string
    common_operations:
      - name: string
        description: string
        command_template: string  # CLI: "supabase projects api-keys --project-ref ${project_ref}"
        required_inputs:
          - name: string
            type: string
        input_resolution_order:   # ["USER_PROFILE.facts", "ENV", "Owner"]
          - string
    mcp_compat:                    # MCP 互換レイヤー（将来の mcp_resource/tool として export 可能な形）
      resource_type: string | null
      tool_schema: object | null
```

### A4: capability-preflight.md ルール作成

`.claude/rules/capability-preflight.md`:

```markdown
# Capability Preflight — Iron Law

> Owner に技術的な作業を依頼する前に、必ず CAPABILITIES を探索する。

## Iron Law

Manager は Owner に手順依頼をする前に、**必ず** 以下を実行:

1. 依頼のカテゴリ分類（CLI/API/MCP で代替可能か判定）
2. CAPABILITIES.yaml から該当 capability を検索
3. 該当あり + status=available + auth_status=verified なら **自動実行**
4. 該当あり + auth_status=unverified なら **Owner に auth 確認のみ依頼**
5. 該当なし → 初めて手順依頼する

## 違反例（Red Flag）

- Owner に「Supabase ダッシュボードで...」と依頼する前に `CAPABILITIES.yaml` を確認していない
- `which supabase` を毎回実行している（manifest を使っていない）

## 運用

- 毎 Tick 開始時に scan.sh を実行（scripts/capabilities/scan.sh）
- scan.sh は冪等。既存 manifest を尊重しつつ新規発見を追加
- risk_level=high 以上の操作は owner_approval_required_for を必ず参照
```
