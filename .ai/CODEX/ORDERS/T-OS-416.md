## Background

`pretool_policy.py:266` の bypass 実装が単純な string match:

```python
integrator_bypass = "ORGOS_INTEGRATOR=1" in command
if git.subcmd in {"commit", "push"} and not integrator_bypass:
    violations.append(("IntegratorOnlyCommit", ...))
```

Manager 検証で **3/3 bypass exploitable**:

```
Test 1: ORGOS_INTEGRATOR=1 git commit -m bypass     → exit 0 (intended bypass; OK の意図)
Test 2: echo "ORGOS_INTEGRATOR=1"; git commit       → exit 0 (UNINTENDED — string がどこにあっても素通り)
Test 3: git commit -m "...ORGOS_INTEGRATOR=1..."    → exit 0 (UNINTENDED — commit message 内でも素通り)
```

## アーキテクチャ修正方針

GPT-5.5 5th round Q31 / Q24 補足の指摘:
> pretool は raw `git commit` を **env に関係なく拒否**し、`ORGOS_INTEGRATOR=1` は git hook 側で integrator script の child process だけを通すための内部印に限定すべき。

**根本誤り**: 実際には integrator-commit.sh の内部 `git commit` は subprocess (Manager の Bash tool 呼び出し `bash scripts/org/integrator-commit.sh T-XXX` の child process)。subprocess は Claude Code の pretool 対象外。

**結論**: pretool 側の bypass 判定は不要・有害。**完全削除**する。

| ケース | 旧動作 | 新動作 |
|---|---|---|
| Manager Bash: `git commit -m x` | deny (no bypass string) | deny |
| Manager Bash: `ORGOS_INTEGRATOR=1 git commit` | **allow (string match で素通り)** ← BUG | **deny** |
| Manager Bash: `git commit -m '...ORGOS_INTEGRATOR=1...'` | **allow (string match で素通り)** ← BUG | **deny** |
| Manager Bash: `bash scripts/org/integrator-commit.sh T-XXX` | allow (not git command) | allow (not git command) |
| ↑ 内部の `git commit` subprocess | pretool 対象外 | pretool 対象外 |

つまり integrator-commit.sh の通常運用は影響なし。bypass token は dead code として除去。
