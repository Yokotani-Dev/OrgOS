## Acceptance Criteria

### A1: proposal schema
`.claude/schemas/evolution-proposal.yaml`:
- proposal_id (P-YYYY-MM-DD-NNN)
- source_events (array of event_id refs)
- problem_class (P0 / P1 / P2)
- proposed_change (target_file, before/after diff or description)
- rationale
- estimated_blast_radius (silent / local / shared / external)
- estimated_risk_level (low / medium / high / critical)
- autonomy_recommendation (silent_execute / execute_with_report / ask_before_execute / owner_only)
- iron_law_check (forbidden patterns へ抵触しないか self-check)
- reviewer_a (synthesizer LLM)
- reviewer_b (independent LLM, fixture stub OK)
- agreement (agree / disagree)
- escalation_target (null or D-XXX decision card id)

### A2: synthesize.sh
- 引数: 期間 or event filter (例: `--last 7d`, `--problem-class P0`)
- output: `.ai/EVOLUTION/proposals/P-YYYY-MM-DD-NNN.yaml`
- 実装: LLM 呼び出しは fixture stub で OK (本タスクでは「synthesize がプロセスとして動く」が目的)
- 必須 Iron Law check: target_file が `.claude/rules/rationalization-prevention.md` `.claude/rules/request-intake-loop.md` などの forbidden path なら proposal を REJECT

### A3: peer-review.sh
- 引数: proposal_id
- output: proposal の `reviewer_b` と `agreement` を埋める
- disagreement なら `escalation_target` に Decision Card id をセット → `scripts/inbox/add-decision.sh` を call

### A4: 検証
- synthesize.sh で events.jsonl から ≥1 proposal 生成可能
- proposal が schema 準拠
- peer-review.sh が agreement / disagreement を判定
- forbidden Iron Law file への提案は REJECT される (test fixture)
