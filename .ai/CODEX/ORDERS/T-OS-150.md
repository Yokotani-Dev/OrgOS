## Context

### 背景
ChatGPT Pro レビュー（`.ai/DESIGN/CHATGPT_PRO_REVIEW_2026-04-18.md`）で **最大の盲点** として指摘された:

> 「Manager の評価関数がない」 — OrgOS は自律駆動や Iron Law を持つが、「気が利く Manager」の品質を測る指標がない。

これを埋めるための **Manager Quality Eval regression suite** を実装する。
改善効果の測定基盤として Phase 1-4 の実装前に完成させる必要がある。

### 評価する 6 指標（Pro 提示）
```yaml
manager_quality_metrics:
  repeated_question_rate:             # 同じことを二度聞く率
    target: "< 5%"
  owner_delegation_burden:            # Owner に振られる作業量のトレンド
    target: "downward trend"
  context_miss_rate:                  # 文脈を外した応答の率
    target: "< 3%"
  unnecessary_owner_question_rate:    # CLI/memory で解決できたのに聞く率
    target: "< 10%"
  capability_reuse_rate:              # 登録済み手段の再利用率
    target: "> 80%"
  decision_trace_completeness:        # 判断の追跡可能性
    target: "> 95%"
```

特に `repeated_question_rate` と `context_miss_rate` は **P0**。

### Owner の実不満（Eval の原材料）
`.ai/DESIGN/ORGOS_TOBE.md` Part 2 の 3 症状:
- 症状 A: 短絡的な負担転嫁（CLI できるのに GUI 依頼、毎回認証情報を聞く）
- 症状 B: プロジェクト全体像の欠如（並列タスクで文脈飛び、単発依頼で文脈無視）
- 症状 C: 情報補完の姿勢が受動的（推測で埋める、巻き込みが下手）

### A2: 20 ケースの regression suite 設計
各 case は YAML で以下を定義:
```yaml
id: "MQ-001"
category: "repeated_question"         # 6 指標のいずれか
symptom: "症状 A: 短絡的な負担転嫁"
scenario: |
  Owner が 2026-03-15 に supabase project_ref を共有した。
  今回 DB 接続設定を依頼したとき、Manager が再度 project_ref を聞くかどうかを測る。
input:
  past_qa: [{q: "Supabase project_ref?", a: "abc123", asked_at: "2026-03-15"}]
  current_request: "DB 接続設定して"
expected_behavior:
  - "past_qa を参照して再度聞かない"
  - "supabase CLI で自動実行"
  - "応答冒頭で「USER_PROFILE を参照」を明示"
anti_pattern:
  - "DB URL と認証情報を教えてください"
  - "Supabase のどのプロジェクトですか"
metric: "repeated_question_rate"
weight: 1.0
```

**20 ケースの内訳（バランス）**:
- repeated_question 系: 4 ケース（症状 A）
- cli_over_gui 系: 4 ケース（症状 A）
- context_miss 系: 4 ケース（症状 B、並列タスク文脈飛び含む）
- unnecessary_question 系: 3 ケース（症状 C）
- capability_reuse 系: 3 ケース（症状 A & B）
- decision_trace 系: 2 ケース（記録不全）

### A3: metrics.yaml の定義
```yaml
metrics:
  - id: repeated_question_rate
    description: "同じことを二度聞く率"
    formula: "count(questions in past_qa asked again) / count(total questions)"
    target: "< 5%"
    priority: P0
    weight: 2.0
  - id: context_miss_rate
    description: "文脈を外した応答の率"
    formula: "count(responses not binding active work graph) / count(total responses)"
    target: "< 3%"
    priority: P0
    weight: 2.0
  - id: unnecessary_owner_question_rate
    description: "CLI/memory で解決できたのに Owner に聞く率"
    formula: "count(questions answerable by CLI/memory) / count(total questions to owner)"
    target: "< 10%"
    priority: P1
    weight: 1.5
  - id: capability_reuse_rate
    description: "登録済み手段の再利用率"
    formula: "count(requests solved by CAPABILITIES.yaml) / count(total technical requests)"
    target: "> 80%"
    priority: P1
    weight: 1.5
  - id: owner_delegation_burden
    description: "Owner に振られる作業量のトレンド"
    formula: "7日移動平均(owner_tasks_per_day)"
    target: "downward trend"
    priority: P1
    weight: 1.0
  - id: decision_trace_completeness
    description: "判断の追跡可能性"
    formula: "count(decisions with full trace) / count(total decisions)"
    target: "> 95%"
    priority: P2
    weight: 1.0
```

### 設計方針
- **Iron Law**: 20 ケースは Owner の実不満から機械的に導出する。抽象的な「良い Manager」基準を作らない

- **拡張性**: 新ケース追加が YAML 1 ファイルで済む設計
