## Context

OrgOS の現状: DESIGN フェーズは ARCHITECTURE / API_CONTRACT / UI_DESIGN / TECH_RESEARCH のみ要求。
**脅威 / データモデル全体 / 権限境界 / トランザクション境界 / 冪等性** は要求されていない。

結果: 実装後のセルフレビューで「Race condition」「重複 insert」「権限漏れ」「景表法違反コピー」などが
**永遠と発掘される**。観点を絞ったレビューを順番に当てるため、そのレビューが見ていない観点は次のレビューで初めて出る。

これを構造的に止めるには **DESIGN フェーズで「想定脅威の網羅リスト」「データモデル全体図」「権限境界マップ」を必須成果物化**し、
セルフレビューを「漏れチェック」に格下げする (実装前に観点を fix する)。

## Acceptance Criteria

### A1: 新規 Rule 作成
`.claude/rules/pre-implementation-risk-profile.md` を新規作成。以下を含む:

- Iron Law section (5 項目以上):
  - DESIGN フェーズ完了の定義に THREAT_MODEL / DATA_MODEL_FULL / AUTHORITY_BOUNDARY 3 文書必須
  - 脅威モデルなしの IMPLEMENTATION 着手禁止
  - データモデル全体図なしの個別テーブル設計禁止
  - 権限境界 (RLS / RPC 権限 / 認証) を明示せずに認証コード書くこと禁止
  - トランザクション境界と冪等性 key の事前決定必須

- Threat Categories (網羅必須):
  - Race condition (同時更新)
  - 重複 (idempotency 不在)
  - 権限漏れ (RLS / authz)
  - 認証回避 (authn)
  - データ漏洩 (PII / secret)
  - 暴走 (無限ループ / リソース枯渇)
  - 入力検証欠落 (injection / XSS)
  - エラー隠蔽 (silent failure)

- DESIGN Gate 接続 (3 文書 confirmed なしで IMPLEMENTATION 不可)
- Self-Review Reframing (実装後セルフレビュー = 観点漏れ chk、観点自体は実装前に fix)
- Red Flags (10 項目)
- Violation Detection / Response
- Relationship To Other Rules

### A2: design-documentation.md の強化
既存 `.claude/rules/design-documentation.md` に以下を追加:

- 必須ドキュメントタスクに以下を追加:
  - T-DESIGN-THREAT (脅威モデル)
  - T-DESIGN-DATAMODEL-FULL (全テーブル + ER + 状態遷移 + 不変条件)
  - T-DESIGN-AUTHORITY (権限境界マップ)
- プロジェクト種別判定ルールを更新 (regulated/transactional/multi-user では 3 文書必須)
- /org-tick での自動実行に新文書生成を追加

**注意**: 既存セクションは破壊しない。追加のみ。

### A3: THREAT_MODEL.md template
脅威カテゴリごとに:
- カテゴリ名
- 該当箇所 (どの flow / table / endpoint で発生し得るか)
- 攻撃シナリオ (具体例)
- 対策 (実装で何をするか)
- 検証方法 (テストでどう確認するか)

8 カテゴリ全て + プロジェクト固有脅威セクション。

### A4: DATA_MODEL_FULL.md template
- 全テーブル一覧 (purpose / owner / lifecycle)
- ER 図 (mermaid)
- 状態遷移図 (主要 entity の state machine)
- 不変条件 (invariants) リスト
- トランザクション境界 (どの操作が atomic でなければならないか)
- 冪等性 key (どの操作が idempotent で、key は何か)
- 削除戦略 (hard delete / soft delete / archive)

### A5: AUTHORITY_BOUNDARY.md template
- ロール一覧 (system_admin / authenticated_user / anonymous など)
- リソースごとの権限マトリクス (CRUD x role)
- RLS policy 一覧 (Postgres / Supabase 想定)
- RPC / Function 権限一覧
- 認証境界 (どこから先が authenticated 必須か)
- セッション管理 (token life / revoke 戦略)
- 監査ログ要件

### A6: 検証
- 全 5 ファイル作成
- 既存 design-documentation.md は追加のみで破壊なし
- 各 template が空欄を埋める形式で実用的
- Iron Law が「脅威モデルなしの IMPLEMENTATION 着手禁止」を deterministic に表現
