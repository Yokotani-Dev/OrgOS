T-OS-352 (Domain Constraint) と T-OS-353 (Pre-Risk Profile) で確立した観点を、DESIGN フェーズで **並列専門家視点** から集約する agent 群を新設する。

各 agent は独立に分析結果を返し、Manager が集約して Owner に Decision Brief 提示する。

## Acceptance Criteria

### A1: 4 つの agent prompt 作成

各 agent は Markdown frontmatter + 本文の構造。既存 agent (例: `.claude/agents/org-architect.md`, `.claude/agents/org-reviewer.md`) を読んで構造を踏襲する。

各 agent prompt は以下を含む:

- **Role**: 専門家としての位置付け (例: "Threat modeling specialist focusing on race conditions, idempotency, authorization, data leakage, runaway, input validation, and silent failures")
- **Inputs**: 何を読んで分析するか (BRIEF, JOURNEYS, ARCHITECTURE, API_CONTRACT)
- **Output Artifact**: どの template を埋めて出力するか (T-OS-352/353 で作成された templates を参照)
- **Tools**: Read / Grep / Glob (write は対応 template のみ)
- **Iron Law**: 専門範囲を超える分析を返さない、判断には根拠 (file ref) を付ける、不明点は assumption として明示

#### org-domain-analyst.md
- Inputs: BRIEF.md, .ai/RESOURCES/ (調査資料), 既存 DOMAIN_ANALYSIS draft
- Output: `.ai/DOMAIN_ANALYSIS.md` (T-OS-352 template ベース) を埋める
- 専門領域: 法令、業界 policy、platform policy の特定 (内容化はしない、references のみ)
- 必須: knowledge_gaps を明示し、解消には Owner 確認 or WebSearch tasks を提案
- Tools: Read, Grep, Glob, WebSearch

#### org-threat-modeler.md
- Inputs: ARCHITECTURE, API_CONTRACT, DATA_MODEL_FULL (draft でも可)
- Output: `.ai/DESIGN/THREAT_MODEL.md` (T-OS-353 template ベース)
- 専門領域: 8 カテゴリ (Race / 重複 / 権限漏れ / 認証回避 / データ漏洩 / 暴走 / 入力検証欠落 / エラー隠蔽)
- 各 threat category で「該当箇所」「攻撃シナリオ」「対策観点」を列挙
- Tools: Read, Grep, Glob

#### org-data-modeler.md
- Inputs: ARCHITECTURE, API_CONTRACT, JOURNEYS
- Output: `.ai/DESIGN/DATA_MODEL_FULL.md` (T-OS-353 template ベース)
- 専門領域: 全テーブル ER + 状態遷移 + invariants + transaction 境界 + idempotency key + 削除戦略
- 必須: ER は mermaid で記述、不変条件は formal な assertion 形式
- Tools: Read, Grep, Glob

#### org-security-architect.md
- Inputs: ARCHITECTURE, API_CONTRACT, DATA_MODEL_FULL, JOURNEYS
- Output: `.ai/DESIGN/AUTHORITY_BOUNDARY.md` (T-OS-353 template ベース)
- 専門領域: ロール × リソース 権限マトリクス、RLS policy、RPC 権限、認証境界、セッション管理、監査ログ
- 必須: RLS policy は Postgres/Supabase syntax で記述、権限矛盾 (例: anonymous で write 可能だが authenticated で read 不可) を必ず検出
- Tools: Read, Grep, Glob

### A2: Dispatcher Rule
`.claude/rules/specialist-subagents.md` を新規作成:

- 4 agent の起動条件 (DESIGN フェーズで quality_level >= mvp の場合)
- 並列起動の手順 (Manager が同一メッセージで 4 Task tool 呼び出し)
- 集約 protocol (各 agent の output artifact を Manager が読み、Owner に Decision Brief 提示)
- 各 agent の selection ロジック (project 種別 → 起動する agent サブセット)
  - regulated domain → 4 全て
  - transactional system → threat-modeler + data-modeler + security-architect
  - read-only public site → threat-modeler のみ
  - internal tool → security-architect のみ
- Iron Law: agent 自己報告を信用しない、Manager は 4 output artifact を verify

### A3: 検証
- 4 agent prompt 全て作成
- 1 dispatcher rule 作成
- 各 agent が独立 (chain なし)、output artifact が明確
- Iron Law 各 agent prompt に最低 3 項目
