## Context

OrgOS の現状の問題: セルフレビュー観点が **実装後** に当てられる。これは構造的に以下を生む:

1. 観点 1 を実装後に当てる → CRITICAL 発見 → 修正
2. 観点 2 を実装後に当てる → 観点 1 では見えなかった CRITICAL 発見 → 修正
3. 観点 3 を実装後に当てる → 観点 2 では見えなかった CRITICAL 発見 → 修正
4. (永遠に続く)

これを止めるには **観点を実装前に acceptance criteria に書く**:

- TASKS.yaml の各 task の `acceptance` 配列に、Codex がチェックすべき全観点を事前に列挙
- Codex は acceptance を満たす形で実装 (実装後の発見ではなく実装前の制約)
- セルフレビューは「acceptance の漏れチェック」に格下げ

これは Quality Contract (T-OS-350) の DoD を、task 単位で acceptance criteria として展開する protocol。

## Acceptance Criteria

### A1: Iron Law Rule 作成
`.claude/rules/acceptance-pre-write.md` を新規作成。以下を含む:

- Iron Law section (5 項目以上):
  - 全 task の acceptance は実装着手前に確定する
  - acceptance なし、または曖昧な acceptance での Codex 委任禁止
  - 実装中に判明した観点は acceptance を更新してから実装続行
  - Codex の自己報告は acceptance との 1:1 対応で verify
  - セルフレビューは acceptance 漏れチェックに限定 (新観点発見は task の不備とみなし、acceptance 追加 + 再実装)

- Acceptance Quality Standards (acceptance 自体の品質基準):
  - 検証可能 (測定/観察可能な形)
  - 排他的 (重複なし)
  - 網羅的 (Quality Contract DoD の各軸を覆う)
  - 数値化可能なら数値化

- Acceptance Source Mapping (どこから観点を引いてくるか):
  - Quality Contract DoD (functionality / error_handling / security / performance / observability / documentation)
  - Threat Model (T-OS-353 の 8 カテゴリ)
  - Data Model (invariants, transactions, idempotency)
  - Authority Boundary (RLS / authz)
  - Domain Constraint (prohibited / required practices)
  - Journey (happy_path / error_paths)

- Pre-Write Workflow (Manager の作業手順):
  1. task を作成する前に、上記 6 sources から該当観点を抽出
  2. acceptance に展開
  3. Codex に渡す前に Manager 自身でチェック (acceptance が DoD を覆うか)
  4. Codex への Work Order に acceptance を必ず転記

- Self-Review Reframing:
  - 旧: 実装後に「観点を当てる」 → CRITICAL 発見ループ
  - 新: 実装前に「観点を acceptance に書く」 → セルフレビューは漏れチェック
  - 新観点が発見された場合: それは「task の不備」であり、acceptance を更新 + 再実装

- Red Flags (10 項目程度)
- Violation Detection / Response
- Relationship To Other Rules

### A2: ACCEPTANCE_CHECKLIST template

- Task ID

- Threat Categories ごとに「該当する脅威と対策観点」セクション (8 カテゴリ)

各セクションには「該当なし」「N/A」を明示できる field を入れる (空欄は禁止、判断したことを明示)。
