## Acceptance Criteria

### A1: session-bootstrap.md ルール新設
`.claude/rules/session-bootstrap.md`:

```markdown
# Session Bootstrap Protocol — Iron Law

## 目的
OrgOS は「タスク管理 OS」ではなく「Chief of Staff」。
新規セッションでも既存セッションでも、**全ての依頼受付前に** OrgOS の Work Graph / Memory / Capability にバインドする。

## Iron Law
以下は例外なし:

### 1. セッション起動時の強制読込
新規セッションで最初の Manager 応答を生成する前に、以下を全て読む:
- `.ai/USER_PROFILE.yaml` (Safe Memory)
- `.ai/CAPABILITIES.yaml` (Tool manifest)
- `.ai/GOALS.yaml` (Vision / Milestone / Project 階層)
- `.ai/CONTROL.yaml` (プロジェクトスコープ + 権限)
- `.ai/TASKS.yaml` (進行中タスク)
- `.ai/DASHBOARD.md` (現状)
- `.claude/rules/request-intake-loop.md` (最高位 Iron Law)

### 2. 依頼受付時の強制バインド
どんな単発依頼でも、応答前に:
- request-intake-loop.md Step 1-10 を適用
- Step 2 (Load Memory): USER_PROFILE.facts 参照
- Step 3 (Bind Work Graph): 進行中タスクとの関連を判定
- Step 4 (Discover Capabilities): CAPABILITIES 探索
- Step 10 (Report): Coherence mode (Silent/Brief/Full) で回答

### 3. 整合性チェック
依頼内容が以下のいずれに該当するか判定:
- (a) 進行中タスク (TASKS.yaml running) の一部 → 該当タスクに紐付け
- (b) 進行中プロジェクト (GOALS projects) と整合 → サブタスク化提案
- (c) 既存プロジェクトと無関係 → 新規プロジェクト提案 or スコープ外として Owner 確認

### Red Flags (即停止条件)
- USER_PROFILE 未読で応答生成
- GOALS/TASKS 未読で新規依頼を処理
- 単発依頼を進行中プロジェクト文脈から切り離して処理

### Manager が実行すべき手順
1. scripts/session/bootstrap.sh を (擬似) 実行 — 必要ファイルの存在確認 + Summary 出力
2. 結果を handoff_packet.verification に記録
3. 以降の応答は request-intake-loop に従う
```

### A5: Bootstrap 実行順序の定義
session-bootstrap.md に:

```markdown
## Bootstrap 実行順序

1. **Control plane** (CONTROL.yaml + DASHBOARD.md) — プロジェクトコンテキスト取得
2. **Memory** (USER_PROFILE.yaml) — Owner 資産
3. **Capabilities** (CAPABILITIES.yaml) — 利用可能な手段
4. **Goals** (GOALS.yaml) — 長期ビジョン
5. **Tasks** (TASKS.yaml) — アクティブ作業
6. **Recent Decisions** (DECISIONS.md 直近 20 エントリ) — 最近の判断
```

### A6: 依頼受付時の判定フロー
session-bootstrap.md に:

```markdown
## 依頼受付時の判定フロー

依頼原文 → Bootstrap 済みか確認 → 未なら実行
↓
依頼分類:
  if 進行中タスクのキーワード含む → bind to task
  elif 進行中プロジェクトと整合 → propose sub-task
  elif 未登録領域 → propose new project / confirm scope
  else (小規模 ad-hoc) → execute with silent bind
```
