## Context

OrgOS には自律進化を担う 9 機構が存在するが、9 機構中 動 1 / 部分動 4 / 不動 4 で **時計が止まっている** 状態。`/org-evolve` は半年で 1 commit、`daily-health-check.sh` は 1 回のみ実行、`.ai/INTELLIGENCE/` は空、`.ai/OIP/` は 3 ヶ月停止。

本タスクは **Phase 2 Self-Evolution Engine の入口**。既存機構を read-only で統合し、検出されたイベントを統一 schema で `.ai/EVOLUTION/events.jsonl` に蓄積する **Detector v1** を実装する。

既存資産 (read-only で参照):
- `scripts/evolve/daily-health-check.sh` — 既存の health check
- `scripts/evolve/generate-fix-task.sh` — regression からの Work Order 生成
- `scripts/eval/manager-quality-runner.sh` — 20 cases eval
- `scripts/capabilities/scan.sh` — capability scan
- `scripts/memory/normalize-lint.sh`, `promote-lint.sh`, `check-no-plain-secrets.sh`
- `.ai/OIP/` 配下の OIP ファイル (status / age 検出用)
- `.ai/INTELLIGENCE/config.yaml` — watch topics

詳細設計参照: `.ai/REVIEW/T-OS-300/SYNTHESIS.md` §2 と `.ai/REVIEW/T-OS-300/CODEX_ANALYSIS_v2.md` §2.2

## Acceptance Criteria

### A1: Schema 定義
`.claude/schemas/evolution-event.yaml` を新規作成、以下フィールドを必須化:
```yaml
event_id: "EVO-{YYYY-MM-DD}-{NNN}"
detected_at: ISO8601
source: enum [internal_scanner, eval_scanner, capability_scanner, oip_scanner, memory_scanner, intel_scanner]
event_type: enum [eval_regression, ux_drift, capability_new, capability_degraded, rule_stale, iron_law_violation, owner_friction, oip_stale, intel_stale, dna_drift]
severity: enum [P0, P1, P2, P3]
confidence: 0.0..1.0
novelty: enum [first_seen, recurring, transient]
target_artifacts: [{path, lines}]
evidence: [{kind, snippet}]
proposed_action: enum [deduplicate, rename, add, remove, update, escalate]
estimated_impact: enum [small, medium, large]
estimated_risk: enum [low, medium, high]
autonomy_candidate: enum [silent_execute, execute_with_report, ask_before_execute, owner_only]
blast_radius: enum [single_file, multi_file, schema_change, iron_law]
recommended_next: string
```

### A2: Detector 実装
`scripts/evolution/detect.sh` を新規作成:
- `--json` フラグで JSONL 出力 ((`.ai/EVOLUTION/events.jsonl` に追記)
- `--stdout` フラグで標準出力に YAML
- `--scanner <name>` で個別 scanner 実行 (eval / capability / oip / memory / intel)
- 既存スクリプトを read-only で呼び出し、出力を schema に正規化
- **書き込み先は `.ai/EVOLUTION/events.jsonl` と stdout のみ**
- 冪等性: 同じ event は同日内なら重複 append しない (既存 jsonl を読んで dedupe)

### A3: Scanner サブモジュール
`scripts/evolution/scanners/` 配下に以下を新規作成:
- `eval-scanner.sh` — Manager Quality eval の最新実行を読み、target miss を event 化
- `capability-scanner.sh` — `.ai/CAPABILITIES.yaml` の `auth_status=expired` や `verified_at` > 7 日を event 化
- `oip-scanner.sh` — `.ai/OIP/` の `status: Draft` で 14 日経過したものを event 化
- `memory-scanner.sh` — `scripts/memory/normalize-lint.sh --json` の出力を event 化
- `intel-scanner.sh` — `.ai/INTELLIGENCE/raw/` の最終更新が 7 日超なら `intel_stale` event 化

各 scanner は read-only、`--json` 単体実行可能。

### A4: Helper Scripts
- `scripts/evolution/list-events.sh` — 最近の event を表形式で表示
- `scripts/evolution/dedupe-events.sh` — events.jsonl の dedupe (同日内 event_id 重複防止)

### A5: 初回実行で動作確認
- `bash scripts/evolution/detect.sh --json` を実行
- `.ai/EVOLUTION/events.jsonl` に **最低 1 件** の event が記録されること
- Phase 1 で挙がった以下のうち最低 5 件を再検出できること:
  - daily-health-check の継続不足 (`event_type: ux_drift` または `intel_stale`)
  - OIP の滞留 (`event_type: oip_stale`)
  - rule 重複 (`event_type: rule_stale` または同等)
  - capability の verified_at が古い (該当があれば)
  - INTELLIGENCE の空状態 (`event_type: intel_stale`)

### A6: ドキュメント
- `.ai/EVOLUTION/README.md` に scanner 一覧 + event 形式 + 拡張方法を記載
- `scripts/evolution/README.md` に各 scanner の役割を記載
