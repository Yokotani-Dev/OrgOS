## Context

OrgOS の REQUIREMENTS フェーズに **User Journey (業務フロー)** を導入。
BRIEF はペルソナ・動機・成功基準を聞くが、「実際の操作手順」が抜けている。
本タスクで JOURNEYS.yaml と関連 Iron Law を基盤として導入する。

## Acceptance Criteria

### A1: schemas/journey.yaml 新規
```yaml
journey:
  id: string                      # J-XXX
  title: string
  persona: string
  related_milestone: string       # M-XXX (optional)
  related_tasks: list             # T-XXX (optional)
  current_flow:                   # As-Is
    - step: string
      pain_point: string          # optional
  target_flow:                    # To-Be
    - step: string
      enabled_by: string          # optional
  happy_path: list[string]
  error_paths:
    - condition: string
      handling: string
  sync_status: enum[draft, confirmed, superseded]
  confirmed_at: date | null
  confirmed_by: string | null
  created_at: date
  updated_at: date
  notes: string
```

### A2: rules/user-journey-sync.md 新規 (Iron Law)
- 目的: 「何を作るか」ではなく「何を実現したいか」を合意
- REQUIREMENTS gate: sync_status=confirmed 必須
- 追加機能依頼: target_flow 変更を伴うなら After Journey 合意必須
- Red Flags: BRIEF 直後 DESIGN 着手、target_flow Owner 未確認の実装、UX 変更で Journey 未更新
- 違反検出: T-OS-312 (/org-tick gate)、T-OS-313 (request-intake-loop)、Manager Quality Eval 連動

### A3: skills/user-journey-mapping.md 新規
- いつ使うか: REQUIREMENTS で draft→confirmed、追加機能で After Journey
- フレームワーク: As-Is聞く → To-Be作る → happy_path 3-5 ステップ → error_paths 2-3 → Owner 合意
- アンチパターン: 機能リスト、画面遷移図、Owner 未確認 confirmed

### A4: JOURNEYS.example.yaml で OrgOS 自身を例示
- J-ORGOS-001 「OrgOS でプロジェクトを立ち上げる業務」
- current_flow / target_flow を Before/After で記述
- sync_status: confirmed (例として確定済み)

### A5: TEMPLATES/JOURNEYS.yaml 新規
