## Acceptance Criteria

### A1: scripts/authority/request-approval.sh 実装
Manager が呼び出す:
```bash
scripts/authority/request-approval.sh \
  --task-id T-OS-XXX \
  --operation edit_manager_md \
  --target .claude/agents/manager.md \
  --summary "リファクタリング" \
  --impact "Tick フロー変更"
```
動作:
1. `.ai/APPROVALS/<UUID>.yaml` に承認リクエスト書き出し (pending 状態)
2. `.ai/OWNER_INBOX.md` に通知追記:
   ```markdown
   ## [APPROVAL REQUEST] T-OS-XXX (YYYY-MM-DD HH:MM)
   - Operation: edit_manager_md
   - Target: .claude/agents/manager.md
   - Summary: リファクタリング
   - Impact: Tick フロー変更
   - Request ID: <UUID>
   - Status: pending
   - Response: `echo "approve <UUID>" >> .ai/OWNER_COMMENTS.md`
   ```
3. 承認待ち UUID を stdout 出力

### A2: scripts/authority/check-approval.sh 実装
```bash
scripts/authority/check-approval.sh --request-id <UUID>
```
動作:
- `.ai/APPROVALS/<UUID>.yaml` を読む
- `.ai/OWNER_COMMENTS.md` で "approve <UUID>" または "reject <UUID>" を検索
- 見つかれば `.ai/APPROVALS/<UUID>.yaml` の status を更新
- stdout: `{"status": "pending|approved|rejected", "responded_at": "..."}`

### A3: scripts/authority/approval-workflow.sh (統合)
ワークフロー全体を司る:
```bash
scripts/authority/approval-workflow.sh \
  --task-id T-OS-XXX \
  --operation ... \
  --wait  # 同期待機モード (テスト用)
```
動作:
1. request-approval.sh で承認依頼
2. --wait なら check-approval.sh を 30 秒ごとにポーリング (最大 24 時間)
3. 非 --wait なら UUID を返して即終了 (Manager が後でチェック)

### A4: タイムアウト処理
authority-layer.md の approval_workflow:
```yaml
timeout: "Owner 不在 24 時間 → silent_execute 以下に downgrade"
```
- 24 時間経過したら APPROVALS/<UUID>.yaml を `expired` にマーク
- emergency ケース (Owner が P0 障害放置) は downgrade ではなく DECISIONS.md に EMERGENCY-XXX 記録

### A5: apply.sh への統合
既存 `scripts/authority/apply.sh` (T-OS-171) が:
1. os-mutation-check.sh → next_step: request_approval の場合
2. → approval-workflow.sh --wait で承認待機
3. approved なら実適用、rejected なら skip

### A6: 検証
```bash
# Dry run: 承認リクエスト作成
scripts/authority/request-approval.sh --task-id T-TEST --operation edit_claude_md --target CLAUDE.md --summary "test" --impact "test"
# → UUID を返す、.ai/APPROVALS/<UUID>.yaml と OWNER_INBOX.md に記録

# Dry run: 承認チェック (まだ pending)
scripts/authority/check-approval.sh --request-id <UUID>
# → {"status": "pending"}

# 疑似承認: OWNER_COMMENTS.md に記録
echo "approve <UUID>" >> .ai/OWNER_COMMENTS.md
scripts/authority/check-approval.sh --request-id <UUID>
# → {"status": "approved"}
```

### A7: ログ
- 全リクエストを `.ai/AUDIT/approval-YYYY-MM-DD.log` に記録
- approved/rejected/expired のライフサイクルをトレース

### A8: rollback 対応
- approved 後の操作が失敗した場合、APPROVAL を `applied_and_failed` にマーク
- backup からの rollback を Manager が手動実行できるように UUID を残す
