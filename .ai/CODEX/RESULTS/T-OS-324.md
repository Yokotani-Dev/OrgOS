## Rationale

The implementation keeps real LLM calls out of scope by using deterministic fixture stubs:

- Reviewer A is populated during synthesis as `fixture-synthesizer-a`.
- Reviewer B is populated during peer review as `fixture-reviewer-b`.
- `--fixture agree|disagree|auto` provides deterministic smoke coverage without changing the future API boundary.

Synthesis rejects protected Iron Law targets before peer review. The rejected proposal is still written with trace data, `status: rejected`, and `iron_law_check.status: rejected` so the rejection is auditable.

## Risk & Rollback

Primary risks:

- `peer-review.sh` disagreement mode mutates `.ai/OWNER_INBOX.md` through the existing inbox script.
- The proposal schema is a repo-local schema style, not a full JSON Schema validator.
- Fixture heuristics are intentionally simple and should be replaced by real A/B LLM review in a later task.

Rollback:

- Remove the new scripts, schema, README additions, and generated proposal YAMLs listed above.
- No shared ledger file was intentionally edited by this task; disagreement escalation was verified in `/tmp`.
