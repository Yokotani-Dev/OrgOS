## Acceptance Criteria

### A1: cross-session-consistency.md 新規作成
```markdown
# Cross-Session Consistency — Iron Law

## Iron Law
依頼を受けたら応答前に必ず以下を実行:
1. bind_request.sh で依頼内容と既存 Work Graph を照合
2. バインド結果を以下 3 カテゴリに分類:
   - **task_continuation**: 進行中 running/review task の延長
   - **sub_task_of_active_project**: 進行中 milestone/project の新規子タスク
   - **new_project**: 既存とは独立 (Owner 確認推奨)
3. 応答冒頭で「【文脈】...」形式で明示
4. 必要なら TASKS.yaml 更新提案 (ただし編集は Manager 権限で)

## 判定ロジック
- keyword 一致 (タスクタイトル/ notes)
- semantic 類似 (簡易: 共通単語数、TF-IDF 相当)
- 参照関係 (DECISIONS.md からの最近言及)

## Red Flags
- 単発依頼を進行中プロジェクト無視で処理 → Iron Law 違反
- バインド結果を応答で明示しない → context_miss_rate 悪化
```

### A2: scripts/session/bind-request.sh 実装
- 引数: 依頼原文 (stdin or `--request`)
- 出力: JSON
  ```json
  {
    "classification": "task_continuation" | "sub_task_of_active_project" | "new_project",
    "related_tasks": [{"id": "T-OS-XXX", "title": "...", "similarity": 0.85}],
    "related_milestones": [{"id": "M-XXX", "title": "..."}],
    "related_decisions": [{"id": "DEC-XXX", "date": "..."}],
    "suggested_action": "bind | propose_sub_task | confirm_new_project",
    "response_prefix": "【文脈】..."  // 応答冒頭に付ける文言
  }
  ```

### A3: similarity 判定ロジック
- 単純実装で OK (bash + awk / python3)
- キーワード一致 score 計算
- タスクタイトル + notes + acceptance との比較
- 閾値: 0.3 で sub_task 候補、0.7 で task_continuation
