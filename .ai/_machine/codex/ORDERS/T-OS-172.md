## Context

authority-layer.md の Role Matrix を実行時に enforce する。
Owner / Manager / Codex / Reviewer が各ファイルに対して許可された操作を機械的に check。

## Acceptance Criteria

### A1: scripts/authority/role-matrix-check.sh 実装
```bash
scripts/authority/role-matrix-check.sh \
  --principal codex_implementer \
  --operation edit \
  --target CLAUDE.md
```
出力 JSON:
```json
{
  "allowed": false,
  "principal": "codex_implementer",
  "target": "CLAUDE.md",
  "operation": "edit",
  "reason": "codex_implementer cannot edit OS core files",
  "required_principal": ["owner", "manager"]
}
```

### A2: scripts/authority/principal-detect.sh 実装
- 環境変数 or 実行コンテキストから principal を判定:
  - `CODEX_THREAD_ID` 設定あり → codex
  - CLI 引数の `--role` 指定
  - デフォルト: manager
- stdout に principal 名を出力

### A3: role-matrix.yaml の解釈
既存の `.claude/schemas/role-matrix.yaml` を parse して:
- 各 principal の can_edit (list of paths / patterns)
- approvals_required
- read_only

path マッチングは glob 対応 (`.ai/**`, `.claude/rules/*.md` 等)。

### A4: os-mutation-check.sh との統合
`scripts/authority/apply.sh` が:
1. principal-detect.sh で principal 特定
2. role-matrix-check.sh で principal × operation × target を check
3. os-mutation-check.sh で allow_os_mutation と照合
4. 両方 OK で実行 → 1つでも fail なら reject

### A5: 違反ログ
- 拒否された操作を `.ai/AUDIT/rbac-YYYY-MM-DD.log` に記録 (JSON Lines)
- principal, operation, target, rejected_reason

### A6: 検証
```bash
# Codex が CLAUDE.md 編集 → reject
scripts/authority/role-matrix-check.sh --principal codex_implementer --operation edit --target CLAUDE.md

# Manager が .ai/USER_PROFILE.yaml 編集 → allowed
scripts/authority/role-matrix-check.sh --principal manager --operation edit --target .ai/USER_PROFILE.yaml

# Reviewer が実装ファイル編集 → reject (read_only)
scripts/authority/role-matrix-check.sh --principal codex_reviewer --operation edit --target scripts/session/bootstrap.sh
```

### A7: graceful fallback
- role-matrix.yaml 読めない時は warning 出して allowed: true (fail-open)
- ただし AUDIT ログに warning 記録

### A8: 共通基盤
- scripts/authority/common.sh (T-OS-171 で作成) を source
- principal 判定ロジックを外部化
