## Context — Phase 1 を踏まえた方向転換

Phase 1 分析 (`.ai/REVIEW/T-OS-300/CLAUDE_ANALYSIS.md` と `.ai/REVIEW/T-OS-300/CODEX_ANALYSIS.md`) で、フォルダ統治・ルール重複・起動時間 等の **戦術的課題** は網羅された。

しかし Owner は方向転換を表明:

> 「起動時間は小粒の論点だから、もっと抜本的にユーザー体験を向上させるような改善をしたい！というか、人に依存しないような改善を継続的に実行できるような仕組みにしたい。AI の進化が日進月歩すぎて、私がこうしたい、というのに依存しているとむしろ進化が遅れると思うんだよね」

**含意**:
- 戦術的な polish (起動時間、フォルダ整理) は **Owner からの指示なしで OrgOS が自動で実行** すべき
- Owner は ボトルネック であってはならない
- AI モデルの進化に追従できる **可塑性** を OS に埋め込む
- 「Owner にこうしたい」を聞きにいく時点でフロー違反

**したがって、Phase 2 では以下を設計する**:
1. **Self-Evolution Engine** — Owner からの指示なしに OrgOS が自分の課題を発見し、修正案を提案し、検証し、適用する loop
2. **Adaptive Capability Layer** — 新しい AI モデル/MCP/CLI に自動適応する仕組み
3. **Owner Bandwidth Conservation** — Owner の認知/時間を可能な限り消費しないインタラクション設計
4. **"OS DNA"** — OrgOS の構成を versioned, diff-able, evolvable な single source of truth として表現

## Required Deliverables

`.ai/REVIEW/T-OS-300/CODEX_ANALYSIS_v2.md` に以下を構造化して書く。

### 1. Diagnosis: なぜ既存自律進化が機能していないか
1.1 既存自律進化機構の棚卸し (上記インフラ × 動作実績)
1.2 各機構の root cause analysis (なぜ Owner が「人に依存している」と感じるか)
1.3 「人手が必要」になる場面の typology (P0=避けられない / P1=避けるべき / P2=避けられる)
1.4 Phase 1 で挙がった課題のうち、自律進化機構があれば人手不要だったものリスト (証拠つき)

### 2. Self-Evolution Engine 設計
2.1 アーキテクチャ全体図 (mermaid 推奨)
2.2 Detection layer — 課題の自動発見
  - eval 失敗、metrics drift、新規 capability 出現、未使用 rule、Iron Law 違反、ユーザフィードバック sentiment
  - 検出 cadence (連続/日次/週次) と判定基準
2.3 Synthesis layer — 改修案の自動生成
  - LLM 駆動 (Claude / Codex) でパッチ案を出す方法、必要な context、cost budget
  - 案の品質保証 (peer-review by another agent, simulated dry-run)
2.4 Validation layer — 修正案の検証
  - eval 回帰テスト、A/B テスト (variant rules)、synthetic Owner simulation
  - 検証コスト と 採用閾値
2.5 Application layer — 適用と rollback
  - autonomy_level に基づく自動適用 vs Owner approval
  - 段階適用 (canary, shadow mode, full)
  - rollback 条件と手順
2.6 Observability — 進化トレース
  - 何がいつなぜ変わったかの記録、Owner への要約レポート

### 3. Owner Bandwidth Conservation Strategy
3.1 Owner が消費する 3 リソース (時間 / 認知 / 意思決定回数) の現状と削減目標
3.2 Owner Touchpoint の typology
  - 真に Owner にしか決められない (Vision, Budget, Compliance) — 残す
  - Owner の好みが必要だが推測可能 (literacy, review_cadence) — 推測 + override 可
  - 単に Owner が知っている情報 (API key, project ref) — 自動取得 or memory
  - Owner に通知するだけ (進捗、結果) — push ではなく pull
3.3 Owner Inbox を「質問の山」から「決済テーブル」へ転換する設計
3.4 デフォルト値の sourcing (community defaults, learned defaults, similar-project transfer)

### 4. Adaptive Capability Layer 設計
4.1 新 LLM 出現時の自動適応 (model upgrade probe, regression test)
4.2 新 MCP/CLI 出現時の自動取り込み (capability scan の trigger と質保証)
4.3 既存 capability の degradation 検知と自動 mitigation
4.4 「AI 進化に伴う OS の変化形跡」を可視化する mechanism

### 5. OS DNA — Self-Describing OrgOS
5.1 単一の DNA file (例: `.ai/ORG_DNA.yaml`) で全 rule / agent / skill / schema / capability を versioned に表現
5.2 Diff 可能な変更履歴 (semver 風)
5.3 他 OrgOS インスタンスとの DNA 同期 (best-of-breed の community 進化)
5.4 DNA に基づく automated regeneration of `.claude/` (DNA → 各ファイル)

### 6. Re-imagined UX (戦術ではなく構造から)
**注意**: 「起動時間短縮」「フォルダ整理」のような戦術的提案は Phase 1 で扱った。Phase 2 では構造的 UX 転換のみ扱う。

6.1 Conversational Sovereignty — Owner が「何をすべきか考える」必要のないインタラクションモード設計
6.2 Result-First UX — 「進捗」ではなく「次の決済ポイント」を起点に並べる DASHBOARD 再設計
6.3 Always-On Mode — Owner が claude code を起動していないときも OS が判断・実行・通知する pathway
6.4 Failure Disclosure — 失敗・遅延・脱線を Owner に「通知」として届け、判断を求める形に
6.5 Marketplace UX — community OrgOS DNA を自プロジェクトに適用する体験

### 7. Implementation Roadmap
7.1 Phase 2 ロードマップ (T-OS-XXX 単位、3-6 ヶ月)
7.2 各タスクの success criteria (測定可能)
7.3 Phase 1 タスクとの依存関係
7.4 リスク分析と mitigation

### 8. Anti-patterns to Avoid
- AI 進化を OS 進化が逆に阻害するパターン
- 自律進化が暴走するパターン (Iron Law 自己改修等)
- Owner を完全排除して逆に satisfaction が下がるパターン

## Instructions

1. Phase 1 の 2 レポートを必ず読み、戦術論との重複を避ける。
2. 既存自律進化インフラを個別に検証 (file 存在 + 直近運用形跡 + 出力ファイルの実体)。
3. 「動いていない」を主張する場合は **動作証拠の不在** で示す (ログがない/出力ファイルが古い/最終実行日が遠い 等)。
4. 設計案は **実装可能** である必要がある。抽象論で終わらせない。
5. Owner の言葉「AI 進化に追従」を真剣に受け止め、**特定モデル名 / API に依存しない設計** を優先する。
