## Context

### 背景
ChatGPT Pro レビューで指摘された `USER_PROFILE.yaml` の **危険な設計** を修正し、fact registry 方式に刷新する。

**v1 の問題（Pro 指摘）**:
- credentials に実体や推測可能な情報を置く設計は危険
- preference と fact と secret と decision が混ざっている
- scope, confidence, source, expires_at, last_verified が不足
- 「覚えていること」が間違っていた場合の修正・失効ルールがない

### 目指す設計（Pro 推奨）
**facts / secrets / preferences を明確に分離**。secret は実体ではなく **secret pointer** を保存。

### 解決する Pro 指摘盲点
- 盲点 1 (PII リスク): secret 実体を YAML に置かない → URI 形式の pointer
- 盲点 3 (記憶ライフサイクル欠如): capture → normalize → scope → retrieve → validate → retire/promote の 6 操作を Iron Law 化

## Acceptance Criteria

### A1: USER_PROFILE.yaml の schema 定義

`.claude/schemas/user-profile.yaml` に schema を定義:

```yaml
# USER_PROFILE.yaml schema
owner:
  name: string
  literacy_level: enum[beginner, intermediate, advanced]
  response_preference: string
  timezone: string

facts:
  - id: string          # fact_<semantic_slug>
    type: enum[project_resource, personal_info, preference, environment, relationship]
    value_ref: any      # 具体値（plain string OK、ただし pii_level に応じて扱い変更）
    scope: string       # "global" | "project:<id>" | "domain:<name>"
    source: enum[owner_confirmed, inferred, auto_detected, external]
    source_ref: string  # 具体的な出典（past_qa:YYYY-MM-DD, file:path#L, etc.）
    confidence: float   # 0.0 - 1.0
    valid_from: date
    expires_at: date | null
    pii_level: enum[none, low, medium, high]
    secret: bool        # true なら secrets セクションへ

secrets:
  - id: string          # secret_<semantic_slug>
    service: string
    scope: string
    storage: string     # URI: 1password://..., keychain://..., sops://..., env://VAR_NAME
    materialized: bool  # 実体を持たない場合 false
    last_verified_at: date
    rotation_policy: enum[manual, auto_monthly, auto_quarterly, never]

preferences:
  - id: string          # pref_<semantic_slug>
    statement: string
    scope: string
    source: enum[owner_confirmed, inferred]
    confidence: float
    overridable: bool
    learned_at: date
```

### A4: memory-lifecycle.md ルール作成

`.claude/rules/memory-lifecycle.md` を新規作成。Iron Law として 6 操作を定義:

```markdown
# Memory Lifecycle — Iron Law

> Memory の操作は以下 6 段階のみ。例外なし。

## Iron Law

USER_PROFILE の各エントリは以下のライフサイクルを辿る:

1. **capture** - 新情報を fact/secret/preference として登録する
   - 必須フィールド全て埋める
   - confidence 不明なら 0.5 以下
   - source を必ず記録

2. **normalize** - 表記ゆれ・重複を統一する
   - 同一 semantic の fact は merge
   - scope を明示化

3. **scope** - 適用範囲を限定する
   - global / project:<id> / domain:<name>
   - 誤転移防止

4. **retrieve** - 依頼処理時に参照する
   - Request Intake Loop の Step 2 で強制
   - scope / expires_at / confidence でフィルタ

5. **validate** - 定期または参照時に妥当性確認
   - expires_at 超過 → 再確認を要求
   - confidence < 0.5 → Owner 確認

6. **retire / promote** - 失効させる or scope 昇格
   - retire: expires_at 超過 or Owner 否定
   - promote: project → domain → global
```

さらに:
- 各操作の具体的手順
- 違反時の対応（Red Flag）
- Manager の強制ステップ（Iron Law 違反検出）

### 設計方針
- **Iron Law**: secret は実体を YAML に置かない。必ず pointer 化
- **scope 必須**: 誤転移防止のため全 fact に scope を付ける
- **expires_at**: 揮発性のある fact（例: 認証トークン有効期限）は expires_at を付ける
- **backward compatibility**: 将来の graph memory (Zep 等) への移行を阻害しない設計
